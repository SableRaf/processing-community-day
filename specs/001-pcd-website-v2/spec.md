# Feature Specification: PCD Website v2.1

**Feature Branch**: `001-pcd-website-v2`
**Created**: 2026-03-03
**Status**: Draft
**Input**: User description: "Build PCD Website v2.1 — static world map site for Processing Community Day 2026 with Leaflet map, node detail panel, alphabetical node list overlay, and full keyboard accessibility"

## Overview

Processing Community Day (PCD) 2026 marks the 25th anniversary of Processing. Events take place across October 2026 as a globally distributed set of independently organized local events, called **nodes**. This website gives the public a single place to discover all confirmed PCD 2026 nodes, explore event details, and save events to their calendars — with no backend, no API keys, and no login required.

The site is a clean rebuild of the existing proof-of-concept. Node data is supplied as a static JSON file generated by the existing data pipeline (Google Form → Sheets → CSV → `nodes.json`). The website consumes that file at build time and publishes a fully static site.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Discover nodes on the map (Priority: P1)

A visitor lands on the site and immediately sees a world map populated with red markers at each PCD 2026 node location. They can pan, zoom, and interact with the map to explore where events are happening globally. At low zoom levels, nearby markers are grouped into clusters that expand on click.

**Why this priority**: The map is the primary interface and the first thing every visitor sees. Without it, the site provides no value.

**Independent Test**: Can be tested by opening the site and verifying that markers appear on the map at correct locations. Delivers the core discovery value independently of panels or list overlays.

**Acceptance Scenarios**:

1. **Given** the site loads, **When** the map finishes rendering, **Then** all confirmed PCD 2026 nodes appear as red circle markers at their correct geographic positions.
2. **Given** multiple nodes are close together, **When** the map is at low zoom, **Then** those nodes are grouped into a cluster marker showing the count; clicking the cluster zooms in and expands it.
3. **Given** the map is displayed, **When** the user pans or zooms, **Then** markers remain correctly positioned and clusters update dynamically.
4. **Given** the map is loading, **When** tiles have not yet arrived, **Then** a loading indicator is shown; it disappears once the first tiles load.
5. **Given** a node has an invalid or missing location code in the data, **When** the site is built, **Then** the build fails with a clear error message identifying the problematic node.

---

### User Story 2 - Read full event details (Priority: P2)

A visitor clicks a marker on the map. A popup appears with the event name, date, venue, a short description, and quick links. They click "Read more" to open a slide-in panel showing the full event description, venue map link, website link, calendar links, and organizer contact.

**Why this priority**: Discovering nodes is only useful if visitors can learn about events and take action (add to calendar, visit the website). This completes the core user journey.

**Independent Test**: Can be tested by clicking any marker, verifying the popup content, then clicking "Read more" to verify the detail panel — independently of the list overlay.

**Acceptance Scenarios**:

1. **Given** the map is loaded, **When** the user clicks a marker, **Then** a popup opens showing the node name, formatted date, venue name and address, a short description, and a "Read more" button.
2. **Given** a popup is open, **When** the user clicks "Read more", **Then** the popup closes and a detail panel slides in from the right showing: node name, full date and location, venue with a link to a mapping service, full description (split into paragraphs), event website link, Google Calendar link, downloadable calendar file link, and organizer email.
3. **Given** the detail panel is open, **When** the user clicks the close button (×), **Then** the panel slides out and focus returns to the map.
4. **Given** a node has no physical address in the data, **When** the detail panel and popup show the venue map link, **Then** the link uses the venue name and city as the search query instead.
5. **Given** the detail panel is open, **When** the user clicks the Google Calendar link, **Then** a correctly pre-filled Google Calendar event opens in a new tab.
6. **Given** the detail panel is open, **When** the user clicks the calendar file download link, **Then** a `.ics` file is downloaded with correct event details for import into any calendar application.

---

### User Story 3 - Browse all nodes alphabetically (Priority: P3)

A visitor clicks the burger menu button to open a full alphabetical list of all PCD 2026 nodes. They can scroll through the list, see each node's name, city, country, and date, and click any entry to fly the map to that node and open its popup.

**Why this priority**: The alphabetical list serves visitors who want to scan all events (e.g., searching for their city) without interacting with the map, and is the primary keyboard interface for node selection.

**Independent Test**: Can be tested by opening the list overlay and clicking any node entry — independently verifiable without testing the map markers.

**Acceptance Scenarios**:

1. **Given** the map is displayed, **When** the user clicks the burger menu button, **Then** the node list overlay opens showing all nodes sorted alphabetically by name, each entry displaying node name, city, country, and formatted date.
2. **Given** the node list is open, **When** the user clicks a node entry, **Then** the list closes, the map flies to that node's location, and that node's popup opens.
3. **Given** the node list is open, **When** the user clicks the close button (×), **Then** the overlay closes and focus returns to the burger menu button.
4. **Given** the node list is open, **When** the user presses Escape, **Then** the overlay closes.

---

### User Story 4 - Navigate entirely by keyboard (Priority: P4)

A keyboard-only user (or a screen reader user) can skip to the map, open the node list with the M key, navigate through the list with arrow keys, select a node, read its full details in the panel, and close the panel — all without touching a mouse.

**Why this priority**: Full keyboard accessibility is a stated requirement of the project and required for compliance with web accessibility standards.

**Independent Test**: Can be tested by disconnecting the mouse and completing the full discovery-to-detail flow using only Tab, M, arrow keys, Enter, and Escape.

**Acceptance Scenarios**:

1. **Given** the page loads, **When** the user presses Tab, **Then** a "Skip to map" link becomes visible and can be activated to move focus directly to the map.
2. **Given** the map is focused, **When** the user presses M (and no text input is focused), **Then** the node list overlay opens.
3. **Given** the node list is open, **When** the user presses the up/down arrow keys, **Then** focus moves between list items.
4. **Given** the node list is open and a node is focused, **When** the user presses Enter, **Then** the list closes, the map flies to that node, and its popup opens.
5. **Given** the detail panel is open, **When** the user presses Escape, **Then** the panel closes and focus returns to the map.
6. **Given** the node list or detail panel is open, **When** the user presses Tab, **Then** focus cycles only within the open overlay (focus is trapped).
7. **Given** the burger menu button is focused, **When** the user activates it, **Then** focus moves to the first item in the node list.

---

### Edge Cases

- **No nodes in data**: If `nodes.json` contains zero nodes, the map renders empty with no markers or clusters.
- **Node with missing optional fields**: If a node lacks an address, venue and city are used as the maps link fallback. If a node lacks a long description, the short description is used in the detail panel.
- **Node with invalid location code**: Build fails immediately with an error identifying the offending node — no silent skip.
- **Node with special characters in name/description**: All text is safely rendered without interpreting it as markup; no cross-site scripting is possible.
- **Very long description**: The detail panel handles arbitrarily long descriptions without breaking the layout.
- **Mobile viewport (≤ 720px)**: The detail panel becomes full-width; the node list overlay becomes full-screen.
- **Detail panel and node list open simultaneously**: Opening one always closes the other; they are mutually exclusive.

## Requirements *(mandatory)*

### Functional Requirements

**Map Display**

- **FR-001**: The site MUST display all confirmed PCD 2026 nodes as markers on a world map at the correct geographic coordinates derived from each node's location code.
- **FR-002**: The map MUST cluster nearby markers at low zoom levels, showing a count of grouped nodes; clusters MUST expand when clicked.
- **FR-003**: The map MUST use a monochrome light style with no points of interest, closely matching the existing proof-of-concept's appearance.
- **FR-004**: The map MUST display a loading indicator while tiles are loading, which disappears once the map is ready.
- **FR-005**: The build process MUST fail with a clear, descriptive error if any node's location code is invalid or is a short/relative code rather than a full global code.

**Marker Popup**

- **FR-006**: Clicking a marker MUST open a popup containing: node name, formatted event date, venue name and address (or venue and city if no address), a short event description, a "Read more" button, a link to the event website, a Google Calendar link, a downloadable calendar file link, and the organizer email.
- **FR-007**: All user-supplied text rendered in popups MUST be safely escaped so that special characters cannot be interpreted as markup.

**Node Detail Panel**

- **FR-008**: Clicking "Read more" in a popup MUST open a slide-in detail panel showing: node name, full formatted date and location (city, country), venue with a link to a mapping service, the full event description rendered as paragraphs, event website link, Google Calendar link, downloadable calendar file link, and organizer email as a mailto link.
- **FR-009**: If the node has no physical address, the mapping service link MUST fall back to using the venue name, city, and country as the search query.
- **FR-010**: The detail panel MUST slide in from the right edge of the screen with a transition animation.
- **FR-011**: The detail panel MUST be full-width on screens 720px wide or narrower.
- **FR-012**: The detail panel MUST have a visible close button that closes the panel.
- **FR-013**: Opening the detail panel MUST close the node list overlay if it is open.

**Node List Overlay**

- **FR-014**: A permanently visible burger menu button MUST be present on the map, which opens a node list overlay showing all nodes sorted alphabetically by name.
- **FR-015**: Each entry in the node list MUST show: node name, city and country, and formatted event date.
- **FR-016**: Clicking a node entry in the list MUST close the overlay, fly the map to that node's location, and open that node's popup.
- **FR-017**: The node list overlay MUST have a visible close button.
- **FR-018**: The node list overlay MUST be full-screen on screens 720px wide or narrower.
- **FR-019**: Opening the node list overlay MUST close the detail panel if it is open.

**Calendar Links**

- **FR-020**: The Google Calendar link MUST pre-fill the event title, date (as an all-day event), location, and a short description.
- **FR-021**: The downloadable calendar file MUST be a valid `.ics` file that can be imported into any standard calendar application, with correct event title, all-day date, location, description, and event URL.

**Keyboard Accessibility & Accessibility Conformance**

- **FR-022**: A "Skip to map" link MUST be present at the top of the page, visible on keyboard focus, allowing users to bypass navigation and jump directly to the map.
- **FR-023**: Pressing M while the map is focused (and no text input field is active) MUST toggle the node list overlay open or closed.
- **FR-024**: Pressing Escape MUST close whichever of the detail panel or node list overlay is currently open.
- **FR-025**: The node list overlay MUST support arrow key navigation between list items.
- **FR-026**: The detail panel MUST trap focus while open — Tab and Shift+Tab MUST cycle only within the panel's interactive elements.
- **FR-027**: The node list overlay MUST trap focus while open — Tab and Shift+Tab MUST cycle only within the overlay's interactive elements.
- **FR-028**: When the detail panel closes, focus MUST return to the map container.
- **FR-029**: When the node list overlay closes, focus MUST return to the burger menu button.
- **FR-030**: On open, focus in the detail panel MUST move to the close button or panel heading.
- **FR-031**: On open, focus in the node list MUST move to the first list item.
- **FR-032**: The burger menu button MUST communicate its state (expanded/collapsed) to assistive technologies.
- **FR-033**: The detail panel and node list overlay MUST each be identified to assistive technologies as dialogs with descriptive labels.
- **FR-038**: The site MUST conform to **WCAG 2.2 Level AA**. This includes: minimum 4.5:1 text contrast ratio, visible focus indicators meeting minimum size/contrast (SC 2.4.11), no dragging-only interactions (SC 2.5.7), and consistent help mechanisms (SC 3.2.6) if any help elements are present.

**Deployment**

- **FR-034**: The site MUST be deployable as a fully static site with no server-side runtime, no database, and no required API keys.
- **FR-035**: The deployed site MUST serve static assets with long-lived cache headers for performance.
- **FR-036**: The site MUST be built with **Astro** in static output mode (`output: 'static'`). All Leaflet and map interaction code MUST run client-side only.
- **FR-037**: The site MUST be deployed to **GitHub Pages** via a GitHub Actions workflow that triggers on push to the main branch. The workflow MUST run `astro build` and publish the `dist/` output directory.

### Key Entities

- **Node**: Represents a single PCD 2026 local event. Key attributes: **unique identifier** (slugified node name + year, e.g. `pcd-berlin-2026`; URL-safe, deterministic, used as ICS UID and anchor key), name, city, country, venue name, optional physical address, event date, geographic location code (full global format), geographic coordinates (decoded from location code at build time using `open-location-code` npm package), event website URL, short description, optional long description, organizer email.
- **Cluster**: A visual grouping of two or more geographically proximate nodes on the map at a given zoom level. Not a stored entity — derived dynamically from node positions and current zoom level.
- **Calendar Event**: A representation of a node suitable for import into a personal calendar. Derived from node data; made available as a pre-fill URL for Google Calendar and as a downloadable ICS file.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: All PCD 2026 nodes appear on the map at their correct locations — zero nodes missing, zero nodes misplaced.
- **SC-002**: The site loads and displays the map with markers in under 3 seconds on a standard broadband connection.
- **SC-003**: A visitor can discover a node, read its full details, and add it to their calendar in under 60 seconds without any prior familiarity with the site.
- **SC-004**: The complete user journey — map → marker click → popup → "Read more" → detail panel → calendar link — is achievable using keyboard alone, with no mouse interaction required.
- **SC-005**: All interactive elements (nodes via the list overlay, detail panel, node list, burger button, links) are reachable and operable by keyboard — verified by completing the full flow without a mouse. The site MUST conform to **WCAG 2.2 Level AA**.
- **SC-006**: No node data (name, description, venue, email) can be rendered in a way that executes arbitrary code in the visitor's browser — verified by testing inputs containing markup characters.
- **SC-007**: The site deploys successfully to the production hosting platform and passes all 18 Definition of Done checks defined in the PRD.
- **SC-008**: The site is usable on mobile devices with screens 720px wide or narrower — detail panel is full-width, node list is full-screen, all controls are reachable by touch.

## Assumptions

- Node data is supplied as a pre-validated `nodes.json` file. The website build does not run data validation scripts — it only checks that location codes are full global format and throws on failure.
- The number of nodes is small (estimated 10–40 for PCD 2026). Performance targets assume this scale; no pagination or lazy loading of node data is required.
- All nodes use location codes in the Open Location Code (Plus Code) format. No other geocoding formats are supported. Decoding at build time uses the **`open-location-code`** npm package (official Google OLC reference library).
- The site targets modern browsers (evergreen Chrome, Firefox, Safari, Edge). No legacy browser support is required.
- Stadia Maps tiles may require a free account registration for production use. If unavailable, CartoDB tiles with labels are an acceptable visual fallback.
- The organizer email field is always present and valid (enforced by the upstream data pipeline).
- The event website URL field is always present and valid (enforced by the upstream data pipeline).
- No analytics, cookie consent, or privacy policy is required for this version.
- No search or filtering of the node list is required — alphabetical ordering is sufficient for the expected node count.
- The build tool is **Astro** (static output mode). Leaflet and all map-related JS run exclusively client-side via Astro's `client:only` directive or equivalent island pattern.
